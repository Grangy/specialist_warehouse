# Система ячеек (location): изменения и подготовка к выгрузке 1С

## Обзор

Ячейка (`location`) — это адрес хранения товара на складе (например: «А-25», «Ч-90», «Склад 3», «Z-10»). Система определяет склад по первой букве ячейки, позволяет при необходимости менять ячейку при сборке/проверке и передаёт актуальные данные в 1С при выгрузке.

---

## 1. Источники данных

### 1.1 Поступление из 1С

При создании или обновлении заказа (POST `/api/shipments`) каждая позиция приходит с полями:
- `location` — ячейка из 1С (может быть пустой)
- `warehouse` — склад от 1С (опционально)

**Определение склада по ячейке** (`warehouseDetector.ts`):
- **Склад 1**: буквы А–П (А, Б, В … П) + позиции без ячейки
- **Склад 2**: буквы Р–Я + зона Z (Z, Z-10 и т.п.)
- **Склад 3**: `location === "Склад 3"`

Если `location` указан, склад берётся из него. Значение склада от 1С используется только при пустом `location`. При расхождении склад из ячейки имеет приоритет.

### 1.2 Хранение в БД

Ячейка хранится в `ShipmentLine.location`:

```
model ShipmentLine {
  id        String   @id
  shipmentId String
  sku       String
  name      String
  qty       Int
  location  String?   // ← ячейка
  warehouse String?   // Склад 1 / Склад 2 / Склад 3
  ...
}
```

`ShipmentTaskLine` ссылается на `ShipmentLine` и наследует `location` через связь `shipmentLine`.

## 2. Изменение ячейки пользователем

### 2.1 Где можно изменить

Ячейку можно менять в двух режимах:
1. **Сборка** — `CollectModal`, `NameModal`
2. **Проверка** — `ConfirmModal`, `NameModal`

Точка входа — клик по ячейке в модалке → режим редактирования → ввод нового значения → сохранение.

### 2.2 Логика сохранения (update-location)

**API:** `POST /api/shipments/[id]/update-location`  
**Тело:** `{ sku: string, location: string | null }`  
**Параметр `id`:** `taskId` или `shipmentId` — оба варианта поддерживаются.

#### Правило: перезапись разрешена

Новое значение всегда сохраняется в БД. Изменения, сделанные на этапе сборки или проверки, уходят в 1С при выгрузке.

#### Ответ API

Успех:
```json
{ "success": true, "message": "Место успешно обновлено", "location": "..." }
```

### 2.3 Поведение на клиенте

1. **Оптимистичное обновление** — UI обновляется сразу при вводе новой ячейки.
2. **Запрос к API** — при сохранении вызывается `update-location`.
3. **`changedLocations`** — хук хранит изменения по `lineIndex` → `location`. При закрытии модалки все несохранённые изменения отправляются в API.


### 2.4 Хуки

- **useCollect** и **useConfirm**:
  - `updateLocation(lineIndex, location)` — обновляет локальное состояние и вызывает `update-location`;
  - при `closeModal` — принудительно сохраняет все записи из `changedLocations` через `update-location`.

---

## 3. Сортировка по ячейкам

В `CollectModal` и `ConfirmModal` позиции сортируются:

1. По статусу (не собрано/не подтверждено → первыми)
2. По наличию ячейки (без ячейки первыми)
3. По значению ячейки (алфавитный порядок)

---

## 4. Подготовка к выгрузке в 1С

### 4.1 Список готовых заказов

**API:** `GET /api/shipments/ready-for-export`

Выборка:
- `status = 'processed'`
- `exportedTo1C = false`
- `deleted = false`

Для каждого заказа формируются финальные данные, включая `lines` с полями:
```json
{
  "sku": "...",
  "name": "...",
  "qty": <подтверждённое кол-во>,
  "location": "<из ShipmentLine>",
  "warehouse": "<из ShipmentLine>",
  ...
}
```

`location` и `warehouse` берутся напрямую из `ShipmentLine` (после возможных обновлений через `update-location`).

### 4.2 Синхронизация с 1С

**API:** `POST /api/shipments/sync-1c`

1. Принимает массив результатов от 1С: `{ id?, number?, customer_name?, success: boolean }`.
2. По `success: true` помечает заказы как `exportedTo1C = true`.
3. В ответе возвращает список заказов, готовых к выгрузке (тот же формат, что и `ready-for-export`).

В ответе каждая позиция содержит:
- `location` — из `ShipmentLine.location`
- `warehouse` — из `ShipmentLine.warehouse`

То есть 1С получает финальные значения ячеек и складов после всех правок пользователя на этапе сборки и проверки.

## 5. Схема потока данных

```
┌─────────────────────────────────────────────────────────────────────────┐
│ 1С (импорт)                                                              │
│ POST /api/shipments → ShipmentLine { location, warehouse }              │
│ warehouse = detectWarehouseFromLocation(location, warehouseFrom1C)       │
└─────────────────────────────────────────────────────────────────────────┘
                                    │
                                    ▼
┌─────────────────────────────────────────────────────────────────────────┐
│ Сборка / Проверка                                                        │
│ Пользователь кликает ячейку → меняет значение                             │
│ updateLocation() → POST /api/shipments/[id]/update-location               │
│   → всегда сохраняем новое значение в БД (перезапись разрешена)           │
│ При закрытии модалки → сохраняем changedLocations                        │
└─────────────────────────────────────────────────────────────────────────┘
                                    │
                                    ▼
┌─────────────────────────────────────────────────────────────────────────┐
│ Выгрузка в 1С                                                            │
│ GET /api/shipments/ready-for-export  → lines[].location                  │
│ POST /api/shipments/sync-1c (ответ) → lines[].location, warehouse       │
│ Данные берутся из ShipmentLine                                            │
└─────────────────────────────────────────────────────────────────────────┘
```

## 6. Важные API и компоненты

| Компонент / API | Роль |
|-----------------|------|
| `POST /api/shipments` | Создание/обновление заказа, `location` из 1С, расчёт `warehouse` |
| `POST /api/shipments/[id]/update-location` | Сохранение ячейки (только при пустой) |
| `GET /api/shipments/ready-for-export` | Список заказов для выгрузки в 1С |
| `POST /api/shipments/sync-1c` | Подтверждение выгрузки и возврат данных для 1С |
| `warehouseDetector.ts` | Определение склада по ячейке |
| `CollectModal` / `ConfirmModal` | Отображение и редактирование ячейки |
| `NameModal` | Редактирование ячейки по клику на позицию |
| `useCollect` / `useConfirm` | Хранение `changedLocations` и вызов `update-location` |

---

## 7. Краткое резюме

1. **Источник при создании** — 1С; склад определяется по первой букве ячейки.
2. **Редактирование** — разрешена перезапись; изменения на этапе сборки/проверки сохраняются и уходят в 1С.
3. **Сохранение** — сразу при вводе и при закрытии модалки для всех изменённых позиций.
4. **Выгрузка в 1С** — `ready-for-export` и ответ `sync-1c` отдают итоговые `location` и `warehouse` из БД.
