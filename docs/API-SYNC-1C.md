# API Синхронизации с 1С

## Endpoint: `POST /api/shipments/sync-1c`

Этот endpoint предназначен для двусторонней синхронизации заказов между системой сборки и 1С.

### Описание

Endpoint выполняет две функции в одном запросе:
1. **Обновление статуса заказов** - помечает заказы как успешно обработанные в 1С
2. **Получение готовых заказов** - возвращает список заказов, готовых к выгрузке в 1С

### Авторизация

Требуется авторизация. Доступ только для пользователей с ролью `admin`.

Поддерживается три способа авторизации:
1. **Заголовки** `X-Login` и `X-Password`
2. **Тело запроса** - поля `login` и `password` в JSON
3. **Cookies** - стандартная авторизация через сессию

### Формат запроса

**С авторизацией через тело запроса (рекомендуется для 1С):**

```json
{
  "login": "admin",
  "password": "admin123",
  "orders": [
    {
      "id": "shipment_id_1",
      "success": true
    },
    {
      "id": "shipment_id_2",
      "success": false
    }
  ]
}
```

**С авторизацией через заголовки:**

```json
{
  "orders": [
    {
      "id": "shipment_id_1",
      "success": true
    },
    {
      "id": "shipment_id_2",
      "success": false
    }
  ]
}
```
(Заголовки: `X-Login: admin`, `X-Password: admin123`)

**Только получение готовых заказов (без обновления статусов):**

```json
{
  "login": "admin",
  "password": "admin123",
  "orders": []
}
```

**Параметры:**
- `orders` (массив, обязательный) - список заказов с результатами обработки в 1С
  - `id` (строка, обязательный) - идентификатор заказа из системы сборки (shipment.id)
  - `success` (boolean, обязательный) - признак успешной обработки заказа в 1С
    - `true` - заказ успешно обработан в 1С, помечается как выгруженный
    - `false` - заказ не был обработан (ошибка в 1С), статус не меняется

**Поля ответа:**

  - Заказы возвращаются только если:
    - Все задания заказа подтверждены (`status = 'processed'`)
    - Заказ еще не был выгружен в 1С (`exportedTo1C = false`)

### Логика работы

1. **Обновление статуса:**
   - Для каждого заказа в запросе с `success: true`:
     - Устанавливается `exportedTo1C = true`
     - Записывается `exportedTo1CAt = текущее время`
   - Заказы с `success: false` не обновляются

2. **Получение готовых заказов:**
   - Система ищет заказы, где:
     - `status = 'processed'` (все задания подтверждены)
     - `exportedTo1C = false` (еще не выгружены в 1С)
   - Для каждого такого заказа формируется полный набор данных (`finalOrderData`)
   - Возвращается массив готовых заказов

### Пример использования

#### cURL

**Пример запроса к серверу `77.222.47.184:3000`:**

```bash
curl -X POST http://77.222.47.184:3000/api/shipments/sync-1c \
  -H "Content-Type: application/json" \
  -H "Cookie: session=your_session_token" \
  -d '{
    "orders": [
      {
        "id": "clx1234567890",
        "success": true
      },
      {
        "id": "clx0987654321",
        "success": false
      }
    ]
  }'
```

**Получить только готовые заказы (без обновления статусов):**

```bash
curl -X POST http://77.222.47.184:3000/api/shipments/sync-1c \
  -H "Content-Type: application/json" \
  -H "Cookie: session=your_session_token" \
  -d '{"orders": []}'
```

#### JavaScript/TypeScript

```typescript
const response = await fetch('/api/shipments/sync-1c', {
  method: 'POST',
  headers: {
    'Content-Type': 'application/json',
  },
  credentials: 'include',
  body: JSON.stringify({
    orders: [
      { id: 'clx1234567890', success: true },
      { id: 'clx0987654321', success: false },
    ],
  }),
});

const data = await response.json();
console.log('Готовые заказы:', data.orders);
```

### Обработка ошибок

**400 Bad Request** - Неверный формат запроса:
```json
{
  "error": "Неверный формат запроса. Ожидается массив orders"
}
```

**403 Forbidden** - Недостаточно прав:
```json
{
  "error": "Недостаточно прав доступа"
}
```

**500 Internal Server Error** - Ошибка сервера:
```json
{
  "error": "Ошибка синхронизации с 1С",
  "details": "Описание ошибки"
}
```

### Важные замечания

1. **Идентификатор заказа:** В запросе используется `id` заказа (shipment.id), а не номер заказа (shipment.number)

2. **Повторная выгрузка:** Заказы с `exportedTo1C = true` не возвращаются в списке готовых заказов

3. **Атомарность:** Обновление статуса и получение готовых заказов происходит в одной транзакции

4. **Пустой массив:** Если в запросе передать пустой массив `orders: []`, система только вернет готовые заказы без обновления статусов

5. **Частичная обработка:** Если некоторые заказы в запросе имеют неверный формат, они пропускаются с предупреждением в логах, но остальные обрабатываются

### Рекомендации по использованию

1. **Периодичность синхронизации:** Рекомендуется вызывать endpoint периодически (например, каждые 5-10 минут) для получения новых готовых заказов

2. **Обработка ответа:** После получения заказов из ответа, обработайте их в 1С и при следующем запросе передайте результаты обработки в поле `success`

3. **Обработка ошибок:** Если заказ не удалось обработать в 1С, передайте `success: false`, чтобы система не помечала его как выгруженный

4. **Логирование:** Рекомендуется логировать все запросы и ответы для отладки и аудита
