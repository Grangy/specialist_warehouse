# 🔒 Логика блокировки сборки

## Обзор

Реализована улучшенная логика блокировки сборки с отслеживанием активности через механизм heartbeat.

## Основные принципы

1. **Активная блокировка** - попап открыт, heartbeat обновляется каждые 5 секунд
2. **Неактивная блокировка** - попап закрыт или пользователь вышел, heartbeat не обновляется более 30 секунд
3. **Перехват блокировки** - возможен только для неактивных блокировок (кроме админа)

## Механизм работы

### 1. Создание блокировки

При открытии модального окна сборки:
- Создается блокировка в БД (`ShipmentTaskLock`)
- Устанавливается `lastHeartbeat = now()`
- Запускается интервал heartbeat (каждые 5 секунд)

### 2. Heartbeat (отслеживание активности)

Пока модальное окно открыто:
- Каждые 5 секунд отправляется POST запрос на `/api/shipments/[taskId]/heartbeat`
- Обновляется `lastHeartbeat` в БД
- Блокировка считается **активной**, если `now() - lastHeartbeat < 30 секунд`

### 3. Проверка активности при попытке перехвата

При попытке другого пользователя заблокировать задание:
- Проверяется существующая блокировка
- Если `now() - lastHeartbeat < 30 секунд` → блокировка **активна** → перехват невозможен
- Если `now() - lastHeartbeat >= 30 секунд` → блокировка **неактивна** → перехват возможен

### 4. Закрытие модального окна

При закрытии модального окна:
- Останавливается интервал heartbeat
- Отправляется запрос на разблокировку (`/api/shipments/[taskId]/unlock`)
- Блокировка удаляется из БД

### 5. Закрытие вкладки/приложения

При закрытии вкладки или приложения:
- Останавливается интервал heartbeat
- Пытается отправить запрос на разблокировку через `fetch` с `keepalive: true`
- Если запрос не отправился, блокировка станет неактивной через 30 секунд

### 6. Скрытие вкладки (visibilitychange)

При скрытии вкладки:
- Heartbeat останавливается
- Блокировка станет неактивной через 30 секунд после последнего heartbeat

При возврате на вкладку:
- Heartbeat возобновляется автоматически

## API Endpoints

### POST `/api/shipments/[taskId]/lock`
Создает или обновляет блокировку задания.

**Логика:**
- Если блокировка существует и активна (heartbeat < 30 сек) → перехват невозможен (кроме админа)
- Если блокировка существует и неактивна (heartbeat >= 30 сек) → перехват возможен
- Если блокировки нет → создается новая

### POST `/api/shipments/[taskId]/unlock`
Удаляет блокировку задания.

### POST `/api/shipments/[taskId]/heartbeat`
Обновляет `lastHeartbeat` для активной блокировки.

### GET `/api/shipments/[taskId]/heartbeat`
Проверяет активность блокировки.

**Ответ:**
```json
{
  "active": true,
  "lastHeartbeat": "2024-12-25T13:23:49.000Z",
  "timeSinceHeartbeat": 5000,
  "timeout": 30000
}
```

## Константы

- `HEARTBEAT_INTERVAL = 5000` (5 секунд) - интервал отправки heartbeat
- `HEARTBEAT_TIMEOUT = 30000` (30 секунд) - таймаут неактивности
- `LOCK_TIMEOUT = 1800000` (30 минут) - максимальное время жизни блокировки

## Схема базы данных

```prisma
model ShipmentTaskLock {
  id           String        @id @default(cuid())
  taskId       String        @map("task_id")
  userId       String        @map("user_id")
  lockedAt     DateTime      @default(now()) @map("locked_at")
  lastHeartbeat DateTime     @default(now()) @map("last_heartbeat")
  task         ShipmentTask  @relation(fields: [taskId], references: [id], onDelete: Cascade)

  @@unique([taskId])
  @@map("shipment_task_locks")
}
```

## Тестирование

Для тестирования логики блокировки:

1. Откройте модальное окно сборки - блокировка должна быть активной
2. Попробуйте открыть то же задание в другой вкладке/браузере - должно быть сообщение об ошибке
3. Закройте модальное окно - блокировка должна быть удалена
4. Подождите 30 секунд после закрытия - блокировка должна стать неактивной
5. Попробуйте открыть задание снова - должно работать

## Особые случаи

### Администратор
Администратор может перехватить любую блокировку, даже активную.

### Сетевые ошибки
Если heartbeat не может быть отправлен (сетевая ошибка), блокировка станет неактивной через 30 секунд после последнего успешного heartbeat.

### Множественные вкладки
Если пользователь открыл несколько вкладок с одним заданием, heartbeat будет отправляться из всех вкладок, но это не проблема - последний heartbeat будет актуальным.
