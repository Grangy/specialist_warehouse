# Миграция на новую систему баллов (только позиции)

## Что изменилось

С 2026-02-02 баллы начисляются **только за количество позиций**. Скорость работы и эффективность больше не влияют на баллы.

### Тарифы баллов за 1 позицию

| Действие | Склад | Баллы |
|----------|-------|-------|
| Сборка | Склад 1 | 1 |
| Сборка | Склад 2, 3 | 2 |
| Проверка сама (выбрал себя) | Склад 1 | 0,78 |
| Проверка сама (выбрал себя) | Склад 2, 3 | 1,34 |
| Проверка с диктовщиком | Склад 1 | проверяльщик 0,39 / диктовщик 0,36 |
| Проверка с диктовщиком | Склад 2, 3 | проверяльщик 0,67 / диктовщик 0,61 |

---

## Источник формул

**Единственный источник:** `src/lib/ranking/pointsRates.ts`. Все расчёты (updateStats, recalc, aggregateRankings) используют его.

⚠️ Не используйте `stats:calculate` и `stats:recalculate-all` — они применяют устаревшую формулу и перезапишут баллы неверно. Эти скрипты откажутся запускаться при наличии `normVersion: positions-only`.

## Инструкция для production

### 1. Резервное копирование (обязательно)

```bash
# Создать бэкап БД перед пересчётом
npm run db:backup
```

### 2. Деплой кода

```bash
git pull origin main
npm install
npm run build
```

### 3. Пересчёт баллов (dry-run)

Сначала проверьте, сколько записей изменится, **без записи в БД**:

```bash
npm run stats:recalc-points
```

В выводе будет: `Изменено записей: N`. Если всё ок — переходите к применению.

### 4. Применение пересчёта

```bash
npm run stats:recalc-points -- --apply
```

Скрипт:
- обновит `TaskStatistics.orderPoints` и `basePoints` по новой формуле;
- пересчитает `DailyStats` из TaskStatistics;
- пересчитает `MonthlyStats` из DailyStats;
- обновит `dailyRank` и `monthlyRank`.

### 5. Перезапуск приложения

```bash
# Если через pm2:
pm2 restart sklad-spec

# Или systemd, docker и т.п. — по вашей конфигурации
```

### 6. Проверка

```bash
# Аудит баллов за последнюю неделю
npm run stats:audit-week
```

В выводе должно быть: `✅ Аудит пройден.` и `Расхождение: 0`.

## Команды

| Команда | Описание |
|---------|----------|
| `npm run stats:recalc-points` | Dry-run пересчёта (без записи) |
| `npm run stats:recalc-points -- --apply` | Применить пересчёт в БД |
| `npm run stats:audit-week` | Аудит баллов за неделю с понедельника |
| `npm run stats:audit-spot-check` | Выборочная проверка (диктовка/проверка/сборка) |
| `npm run stats:audit-user-points -- "Имя"` | Детальный аудит по пользователю |

**Опции audit-spot-check:** `--7` (последние 7 дней), `--month`, `--30`.

## Что делать, если что-то пошло не так

1. **Восстановить БД из бэкапа:**
   ```bash
   npm run db:restore
   ```

2. **Откатить код:**
   ```bash
   git revert HEAD
   npm run build
   pm2 restart sklad-spec
   ```

3. **Если расхождения в аудите:** повторно запустите `npm run stats:recalc-points -- --apply`.

## Производительность топа / бенчмарк

Замер того же пути, что даёт «Загрузка рейтинга» на `/top` (ядро — `aggregateRankings`):

```bash
npm run benchmark:stats
npm run benchmark:stats:quick
```

Опции: `--quick` (только `week`), `--http` (нужен `npm run dev` — полный HTTP-стек). Жёсткие бюджеты для CI: переменные `STATS_BUDGET_MS_TODAY`, `STATS_BUDGET_MS_WEEK`, `STATS_BUDGET_MS_MONTH`.

**Почему долго:** несколько тяжёлых `findMany` по `task_statistics` (десятки тысяч строк) + последовательный расчёт доп. работы по сессиям месяца + публикация баллов тянет те же тяжёлые пути. Цель — **не считать тяжёлое в HTTP-запросе пользователя**, а отдавать готовый снимок или устаревший кэш мгновенно.

### 10 пунктов глобальной оптимизации (по убыванию «эффект / риск»)

1. **Фоновый воркер вместо расчёта в запросе**  
   Раз в N минут (например 5–15) один процесс считает `aggregateRankings` и пишет результат в таблицу `stats_snapshots` или в JSON на диск. HTTP только читает последний снимок. Запросы к `/top` и публичным API **не блокируют** event loop и SQLite.

2. **Жёсткое разделение: «чтение» vs «пересчёт»**  
   Все маршруты (`/api/statistics/top`, `ranking`, `overview`, публичные `user/.../public`) отдают **только кэш/снимок**. Кнопка «обновить» в админке триггерит задачу в очереди, а не 60-секундный `aggregateRankings` в том же процессе.

3. **Увеличить TTL и не инвалидировать весь кэш при каждом деплое**  
   Сейчас кэш в памяти сбрасывается при рестарте. Хранить снимок на диске как **единственный источник для первого ответа** после рестарта + прогрев в `instrumentation` или `pm2` post-start, чтобы первый пользователь не ловил холод.

4. **Свести объём данных в одном проходе БД**  
   Вместо 6+ огромных `findMany` по `task_statistics` с разными фильтрами — **один** запрос с нужными полями и периодом, агрегация в памяти по ролям/датам. Меньше round-trip и меньше дублирования чтения строк.

5. **Индексы и план запросов**  
   `EXPLAIN QUERY PLAN` на самых тяжёлых запросах. Индексы на `shipment_tasks(completed_at)`, `shipment_tasks(confirmed_at)`, связка `task_statistics` ↔ `task` по тем полям, по которым режется период.

6. **Доп. работа: кэш весов и темпа внутри одного пересчёта**  
   `getEfficiencyWeightsForUsers` и `getWarehousePaceLast15Min` не вызывать заново для одинакового 15‑мин ведра и одного набора пользователей — один Map-кэш на весь проход `aggregateRankings`.

7. **Доп. работа: упростить или заранее материализовать баллы сессий**  
   Либо хранить **начисленные баллы** в `extra_work_sessions` при остановке сессии (пересчёт по cron при смене формулы), либо ограничить исторический пересчёт «последние X месяцев» для топа.

8. **Публичный API и rate limit**  
   `/top` и публичные эндпоинты — только чтение снимка; лимиты по IP, чтобы не устраивали DoS повторными полными пересчётами (если что-то всё ещё считается on-demand).

9. **Очередь и один writer к SQLite**  
   Не запускать параллельно несколько полных `aggregateRankings` (разные периоды/фон + пользователь). Один воркер в очереди снимает **database locked** и пики CPU.

10. **Инфраструктура как подстраховка**  
    Пока (1)–(7) не сделаны: `proxy_read_timeout` / `proxy_connect_timeout` для `/api/statistics/`, отдельный pm2-процесс для воркера, мониторинг длительности `benchmark:stats` в CI.

---

**Итог:** чтобы «не ломало весь сайт», расчёт не должен жить в том же потоке, что и отдача страниц. Минимальный быстрый шаг — **(1)+(2)+(3)**; максимальный выигрыш по БД — **(4)+(5)**; доп. работа — **(6)+(7)**.
